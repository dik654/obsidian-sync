## Step 09: R1CS 제약 시스템 — 모든 계산을 곱셈 하나로

### 핵심 질문: ZK 증명의 "언어"란?

```
ZK 증명의 목표:
  "나는 비밀 x를 알고 있고, f(x) = y가 참이다"
  → x를 공개하지 않고 f(x) = y임을 증명

문제:
  f는 임의의 함수 — 덧셈, 곱셈, 조건분기, 해시, ...
  증명 시스템은 어떻게 "임의의 함수"를 다루는가?

답:
  모든 계산을 단순한 형태로 분해:
    a · b = c    (곱셈 하나 = 제약 하나)

  이것이 R1CS — Rank-1 Constraint System
```

> [!important] R1CS의 위치
> ```
> 프로그램/함수 → [R1CS로 변환] → [QAP로 변환] → [Groth16 증명]
>                  Step 09          Step 11         Step 12
>
> R1CS는 "번역기" — 임의의 계산을 증명 시스템이 이해하는 형태로 번역
> 사람이 읽는 코드 → R1CS → 수학(다항식) → 암호학(페어링)
> ```

---

### R1CS란?

```
R1CS = Rank-1 Constraint System

"Rank-1" = 각 제약이 "벡터 내적 · 벡터 내적 = 벡터 내적" 형태
           → 곱셈이 딱 하나 (rank = 1)

하나의 제약:
  (a₁·x₁ + a₂·x₂ + ... + aₙ·xₙ) · (b₁·x₁ + b₂·x₂ + ... + bₙ·xₙ)
  = (c₁·x₁ + c₂·x₂ + ... + cₙ·xₙ)

축약:
  ⟨a, s⟩ · ⟨b, s⟩ = ⟨c, s⟩

여기서:
  s = [1, x₁, x₂, ..., xₙ]  (witness 벡터: "비밀 + 공개 값")
  a, b, c = 상수 계수 벡터
```

> [!abstract] 직관적 이해
> R1CS의 각 제약은 "곱셈 게이트" 하나를 표현한다.
>
> ```
> 일반 프로그래밍:
>   z = x * y          → 곱셈 1번 = R1CS 제약 1개
>   w = x + y          → 덧셈은 "무료"! (선형결합으로 표현)
>   v = x * y + z * w  → 곱셈 2번 = R1CS 제약 2개
> ```
>
> **덧셈은 무료, 곱셈만 비용이 든다** — 이것이 R1CS의 핵심 특성.

### R1CS의 형식적 정의

> [!important] 수학적으로 엄밀한 R1CS 정의
> 직관적 이해 이후, R1CS를 형식적으로 정의한다.
> 이 정의가 QAP 변환(Step 11)과 Groth16(Step 12)의 기초가 된다.

```
정의 (R1CS):
  체 F 위의 R1CS 인스턴스는 다음으로 구성된다:

  - 행렬 A, B, C ∈ F^{m×n}  (m = 제약 수, n = 변수 수)
  - 분할: n = 1 + l + w
    - 인덱스 0: 상수 1 (One)
    - 인덱스 1..l: 공개 입력 (instance)
    - 인덱스 l+1..n: 비공개 입력 (witness)

  witness 벡터 s = (1, x₁, ..., xₗ, w₁, ..., wₘ) ∈ F^n 이
  R1CS를 만족(satisfy)한다 ⟺

    ∀i ∈ {1,...,m}:  (Aᵢ · s) × (Bᵢ · s) = (Cᵢ · s)

  여기서 Aᵢ는 A의 i번째 행벡터, ·는 내적.

  행렬 형태로:
    (A · s) ⊙ (B · s) = C · s
    ⊙ = Hadamard product (원소별 곱)

관계 (Relation):
  R1CS 관계 R은:
    R = {(x, w) : s = (1, x, w)이 모든 제약을 만족}

  여기서 x = 공개 입력 (instance), w = 비공개 입력 (witness)

  ZK 증명 시스템의 목표:
    "나는 w를 알고 있어서 (x, w) ∈ R이다"를
    w를 공개하지 않고 증명
```

> [!note] 형식적 정의와 코드의 대응
> ```
> 수학                          코드
> ─────────────────────────────────────────
> 체 F                        → Fr (BN254 스칼라 필드)
> 행렬 A, B, C                → to_matrices()의 반환값
> 변수 분할 (One/Inst/Wit)    → Variable::One, Instance, Witness
> witness 벡터 s              → ConstraintSystem.values
> 제약 수 m                   → constraints.len()
> 변수 수 n                   → 1 + num_instance + num_witness
> 만족 조건 검사              → is_satisfied()
> ```

### 변수 타입: One, Instance, Witness

```
R1CS의 변수는 세 종류:

1. One (상수 1)
   → 상수를 표현하기 위한 고정 변수
   → 예: "5"를 표현하려면 5 · One

2. Instance (공개 입력)
   → 검증자도 아는 값 (public input)
   → 예: Merkle root, nullifier_hash

3. Witness (비공개 입력)
   → 증명자만 아는 값 (private input = secret)
   → 예: nullifier, secret, merkle_siblings

witness 벡터 s:
  s = [1, instance₁, instance₂, ..., witness₁, witness₂, ...]
       ↑     ↑                          ↑
      One   공개 값들                   비밀 값들
```

> [!note] 왜 One이 필요한가?
> 상수를 선형결합으로 표현하기 위해:
> ```
> "x + 5 = z" 를 R1CS로?
>
> 선형결합: (x + 5·One) · (1·One) = (z)
>           ⟨a, s⟩        ⟨b, s⟩     ⟨c, s⟩
>
> a = [5, 0, ..., 1, ...]  (One에 5, x에 1)
> b = [1, 0, ..., 0, ...]  (One에 1)
> c = [0, 0, ..., 0, ..., 1, ...]  (z에 1)
>
> One이 없으면 "5"를 표현할 방법이 없다!
> ```

### LinearCombination: 선형결합

```
LinearCombination = Σ (계수 × 변수)

예시:
  3·x + 5·y - 2·One
  = [(One, -2), (x, 3), (y, 5)]

코드:
  pub struct LinearCombination(Vec<(Variable, Fr)>);

  사용:
    let lc = LinearCombination::zero()
        + (x, Fr::from_u64(3))     // 3·x
        + (y, Fr::from_u64(5))     // + 5·y
        + (Variable::One, -Fr::from_u64(2));  // - 2
```

> [!tip] 덧셈이 "무료"인 이유
> 덧셈은 선형결합 안에서 흡수된다:
> ```
> x + y를 계산하려면?
>   → 별도의 제약이 필요 없음!
>   → 다음 곱셈의 입력에 (x + y)를 선형결합으로 직접 사용
>
> 예: z = (x + y) * w
>   → 하나의 제약: ⟨[x:1, y:1], s⟩ · ⟨[w:1], s⟩ = ⟨[z:1], s⟩
>   → "x + y" 계산에 제약 0개, 곱셈에 1개 = 총 1개
>
> 반면: z = x * y + w
>   → t = x * y       (제약 1개)
>   → z = t + w        (제약 0개 — z를 선형결합으로 사용)
>   → 총 1개
> ```

### 제약의 구조: ⟨a, s⟩ · ⟨b, s⟩ = ⟨c, s⟩

```
하나의 R1CS 제약:

  A측 (좌변 첫째): LinearCombination
  B측 (좌변 둘째): LinearCombination
  C측 (우변):      LinearCombination

  의미: A(s) × B(s) = C(s)

예: x * y = z
  A = [(x, 1)]
  B = [(y, 1)]
  C = [(z, 1)]

예: (2x + 3) * y = z + 1
  A = [(x, 2), (One, 3)]
  B = [(y, 1)]
  C = [(z, 1), (One, 1)]
```

### 예제 1: 단순 곱셈 x × y = z

```
공개 입력: z (결과)
비공개 입력: x, y (비밀)

witness 벡터: s = [1, z, x, y]
                   ↑  ↑  ↑  ↑
                  One Inst W  W

제약 1개:
  A = [0, 0, 1, 0]  → x
  B = [0, 0, 0, 1]  → y
  C = [0, 1, 0, 0]  → z

검증:
  ⟨A, s⟩ · ⟨B, s⟩ = x · y
  ⟨C, s⟩ = z
  → x · y = z ✓
```

### 예제 2: 피타고라스 정리 x² + y² = z²

```
"비밀 x, y를 알고 있고, x² + y² = z²이다"

x² + y² = z²을 R1CS로 분해:
  → 곱셈이 3번 필요: x², y², z²
  → 하지만! x² + y² = z²에서 "+"는 무료
  → 실제로 필요한 제약은?

방법:
  보조 변수 도입: x_sq = x², y_sq = y²

  제약 1: x · x = x_sq
  제약 2: y · y = y_sq
  제약 3: (x_sq + y_sq) · 1 = z · z
    → 이건 안 된다! 양변 모두 곱셈이 있어야 함
    → 수정: x_sq + y_sq는 선형결합으로 C측에

  실제 분해:
    제약 1: x · x = x_sq           (x² 계산)
    제약 2: y · y = y_sq           (y² 계산)
    제약 3: z · z = x_sq + y_sq    (z² = x² + y²)

  A = [[x:1], [y:1], [z:1]]
  B = [[x:1], [y:1], [z:1]]
  C = [[x_sq:1], [y_sq:1], [x_sq:1, y_sq:1]]

witness 벡터: s = [1, z_sq, x, y, z, x_sq, y_sq]
  (z_sq = z²은 공개 입력이라면 instance, 아니면 witness)

총 3개 제약으로 피타고라스 정리를 표현!
```

> [!important] 보조 변수 (auxiliary variable)
> R1CS에서는 중간 결과를 저장하기 위해 **보조 변수**를 도입해야 한다.
>
> ```
> 일반 코드: result = a * b * c
>
> R1CS 분해:
>   t = a * b      ← 보조 변수 t 도입 (곱셈 1)
>   result = t * c ← 곱셈 2
>
> 2개 제약, 1개 보조 변수
>
> 보조 변수는 witness에 포함되므로 증명자가 값을 제공해야 한다.
> 하지만 검증자에게는 보이지 않는다 (private).
> ```

### 예제 3: 조건문 if b then x else y

```
조건문도 R1CS로 표현 가능:
  result = b * x + (1 - b) * y
  (b는 0 또는 1이어야 함)

분해:

  제약 1: b · (1 - b) = 0          (b가 boolean임을 강제)
    A = [(b, 1)]
    B = [(One, 1), (b, -1)]
    C = [(One, 0)]  = 0

  제약 2: b · x = bx                (보조 변수)
    A = [(b, 1)]
    B = [(x, 1)]
    C = [(bx, 1)]

  제약 3: result = bx + y - b·y     (최종 결과)
    → b · y를 위한 보조 변수 by 필요

  제약 3: b · y = by
    A = [(b, 1)]
    B = [(y, 1)]
    C = [(by, 1)]

  제약 4: result = bx + y - by
    → 이건 순수 선형결합! 제약 불필요.
    → result를 (bx + y - by)로 대체하면 됨

총: 3개 제약 + 2개 보조 변수 (bx, by)
(최적화하면 2개로 줄일 수 있지만, 교육용으로 명시적 표현)
```

> [!abstract] 비선형 연산 → 제약 수 폭증
> ```
> 연산          | R1CS 제약 수
> ─────────────┼──────────────
> 덧셈 a + b    | 0 (무료)
> 곱셈 a * b    | 1
> 거듭제곱 a^n  | ~log₂(n) (square-and-multiply)
> 비교 a < b    | ~254 (비트 분해 필요!)
> 비트 AND      | ~254 (비트 분해 필요!)
> Poseidon hash | ~250
> SHA-256       | ~25,000
> ```
>
> 비트 연산이 비싼 이유: Fr 원소는 254-bit 정수인데,
> 비트 연산을 하려면 254개 비트로 분해 → 각 비트가 boolean 제약
> → Poseidon이 ZK-friendly인 이유 (비트 분해 불필요)

### 산술 회로에서 R1CS로의 컴파일

> [!important] 산술 회로(Arithmetic Circuit)
> 산술 회로는 덧셈 게이트와 곱셈 게이트로 구성된 DAG(방향 비순환 그래프)이다.
> 임의의 다항식 함수를 산술 회로로 표현할 수 있고,
> 산술 회로를 R1CS로 **기계적으로** 변환할 수 있다.

```
산술 회로(Arithmetic Circuit):
  덧셈 게이트와 곱셈 게이트로 구성된 DAG

  ┌──────────────────────────────────────────┐
  │ 산술 회로 → R1CS 변환 알고리즘            │
  │                                          │
  │ 입력: 산술 회로 C                         │
  │ 출력: R1CS 인스턴스 (A, B, C)             │
  │                                          │
  │ 1. 입력 와이어에 변수 할당               │
  │    x₁, x₂, ..., xₙ                     │
  │                                          │
  │ 2. 위상 정렬로 게이트 순회               │
  │                                          │
  │ 3. 각 게이트에 대해:                      │
  │    덧셈 게이트 (a + b = c):              │
  │      → 제약 불필요!                      │
  │      → c를 선형결합 (a + b)로 표현        │
  │                                          │
  │    곱셈 게이트 (a × b = c):              │
  │      → 새 변수 c 할당                    │
  │      → 제약 추가: A=[a], B=[b], C=[c]    │
  │                                          │
  │ 4. 출력 와이어를 instance로 표시          │
  │                                          │
  │ 결과:                                     │
  │   #제약 = #곱셈 게이트                    │
  │   #변수 = #입력 + #곱셈 게이트 (보조변수)  │
  └──────────────────────────────────────────┘

예제: f(x) = x³ + x + 5

  산술 회로:
    t₁ = x · x        (곱셈 게이트 1)
    t₂ = t₁ · x       (곱셈 게이트 2)
    t₃ = t₂ + x       (덧셈 — 무료)
    y  = t₃ + 5       (덧셈 — 무료)

  변수: s = [1, y, x, t₁, t₂]
              ↑  ↑  ↑   ↑   ↑
             One Inst W  Aux  Aux

  제약 (2개만!):
    1. x · x = t₁
       A=[0,0,1,0,0], B=[0,0,1,0,0], C=[0,0,0,1,0]

    2. t₁ · x = t₂
       A=[0,0,0,1,0], B=[0,0,1,0,0], C=[0,0,0,0,1]

  출력 조건: y = t₂ + x + 5
    → (t₂ + x + 5·One) · 1 = y
    → A=[5,0,1,0,1], B=[1,0,0,0,0], C=[0,1,0,0,0]
    (이것도 제약 1개 추가 — 총 3개)

  검증 (x=3):
    t₁ = 9, t₂ = 27, y = 27 + 3 + 5 = 35
    s = [1, 35, 3, 9, 27]

    제약 1: 3 × 3 = 9 ✓
    제약 2: 9 × 3 = 27 ✓
    제약 3: (27+3+5) × 1 = 35 ✓
```

> [!note] 덧셈이 제약 0개인 이유 (심화)
> ```
> 산술 회로에서 덧셈은 "와이어의 합"이다.
> R1CS의 선형결합(LinearCombination)이 바로 이 와이어의 합을 표현한다.
>
> 예: t₃ = t₂ + x
>   → t₃라는 별도의 변수를 만들 필요가 없다!
>   → 다음 제약의 입력에서 (t₂ + x)를 직접 사용
>
> 이것이 R1CS가 "곱셈의 수"에 비례하는 이유:
>   산술 회로의 곱셈 게이트 수 = R1CS 제약 수
>   덧셈 게이트는 선형결합으로 흡수 → 제약 0개
>
> 실용적 함의:
>   회로 최적화 = "곱셈 수 줄이기"
>   예: x⁵ = ((x²)²) · x → 곱셈 3회 (square-and-multiply)
>       x⁵ = x · x · x · x · x → 곱셈 4회 (나이브)
> ```

### 행렬 형태: A · s ⊙ B · s = C · s

```
N개의 제약, M개의 변수가 있으면:

  A: N × M 행렬 (각 행 = 한 제약의 A측 계수)
  B: N × M 행렬 (각 행 = 한 제약의 B측 계수)
  C: N × M 행렬 (각 행 = 한 제약의 C측 계수)
  s: M × 1 벡터 (witness: [1, instance₁, ..., witness₁, ...])

R1CS 만족 조건:
  (A · s) ⊙ (B · s) = C · s

  ⊙ = Hadamard product (원소별 곱)
  즉, 각 행 i에 대해: (A의 i행 · s) × (B의 i행 · s) = (C의 i행 · s)
```

#### 행렬 표현 예: x × y = z

```
변수: s = [1, z, x, y]  (인덱스: 0, 1, 2, 3)

제약 1개: x · y = z
  A행: [0, 0, 1, 0]
  B행: [0, 0, 0, 1]
  C행: [0, 1, 0, 0]

A = [[0, 0, 1, 0]]     1×4 행렬
B = [[0, 0, 0, 1]]     1×4 행렬
C = [[0, 1, 0, 0]]     1×4 행렬
s = [1, z, x, y]       4×1 벡터

검증:
  A·s = [x]
  B·s = [y]
  C·s = [z]
  [x] ⊙ [y] = [xy] = [z] ✓
```

#### 행렬 표현 예: 피타고라스

```
변수: s = [1, x, y, z, x_sq, y_sq]  (인덱스: 0-5)

제약 1: x · x = x_sq
제약 2: y · y = y_sq
제약 3: z · z = x_sq + y_sq

A = [[0, 1, 0, 0, 0, 0],    // x
     [0, 0, 1, 0, 0, 0],    // y
     [0, 0, 0, 1, 0, 0]]    // z

B = [[0, 1, 0, 0, 0, 0],    // x
     [0, 0, 1, 0, 0, 0],    // y
     [0, 0, 0, 1, 0, 0]]    // z

C = [[0, 0, 0, 0, 1, 0],    // x_sq
     [0, 0, 0, 0, 0, 1],    // y_sq
     [0, 0, 0, 0, 1, 1]]    // x_sq + y_sq

s = [1, 3, 4, 5, 9, 16]  (x=3, y=4, z=5의 경우)

검증:
  A·s = [3, 4, 5]
  B·s = [3, 4, 5]
  C·s = [9, 16, 25]
  [3, 4, 5] ⊙ [3, 4, 5] = [9, 16, 25] = [9, 16, 25] ✓
```

> [!tip] 희소 행렬 (Sparse Matrix)
> 실제 R1CS 행렬은 대부분의 원소가 0이다.
> ```
> Poseidon 회로 (~250 제약, ~200 변수):
>   A 행렬: 250 × 200 = 50,000개 원소
>   비영(nonzero) 원소: ~500개 (1%)
>   → 99%가 0!
> ```
> 따라서 sparse 형태 `Vec<(row, col, value)>`로 저장하는 것이 효율적.
> 이것이 `to_matrices()`가 sparse 행렬을 반환하는 이유.

### ConstraintSystem: 제약 시스템 빌더

```rust
/// R1CS 제약 시스템
///
/// 회로를 빌드하는 인터페이스:
///   1. alloc_instance / alloc_witness로 변수 할당
///   2. enforce()로 제약 추가
///   3. is_satisfied()로 모든 제약 만족 확인
///   4. to_matrices()로 (A, B, C) 희소 행렬 추출
pub struct ConstraintSystem {
    // 변수 값들: [1, instance₁, ..., witness₁, ...]
    pub(crate) values: Vec<Fr>,
    pub(crate) num_instance: usize,
    pub(crate) num_witness: usize,
    pub(crate) constraints: Vec<(LinearCombination, LinearCombination, LinearCombination)>,
}
```

#### API 흐름

```rust
// 1. 제약 시스템 생성
let mut cs = ConstraintSystem::new();

// 2. 변수 할당
let x = cs.alloc_witness(Fr::from_u64(3));   // 비공개
let y = cs.alloc_witness(Fr::from_u64(4));   // 비공개
let z = cs.alloc_instance(Fr::from_u64(12)); // 공개

// 3. 제약 추가: x * y = z
cs.enforce(
    LinearCombination::from(x),        // A = x
    LinearCombination::from(y),        // B = y
    LinearCombination::from(z),        // C = z
);

// 4. 검증
assert!(cs.is_satisfied());  // 3 * 4 = 12 ✓
```

trait Circuit의 의미:

    → verify 시: public input으로 증명 검증

### is_satisfied(): 제약 만족 검증

```
is_satisfied()는 "이 witness가 모든 제약을 만족하는가?"를 확인

알고리즘:
  for each constraint (A, B, C):
    a_val = Σ (coeff · values[var_index])  for (var, coeff) in A
    b_val = Σ (coeff · values[var_index])  for (var, coeff) in B
    c_val = Σ (coeff · values[var_index])  for (var, coeff) in C
    if a_val * b_val ≠ c_val:
      return false
  return true

이것은 "native 실행" — 실제로 값을 대입하여 검산
ZK 증명(Groth16)은 이것을 "값을 모르고도 검증"하는 것
```

> [!important] is_satisfied vs Groth16 verify
> ```
> is_satisfied():
>   증명자가 직접 값을 대입하여 검산
>   모든 변수의 값이 필요 (instance + witness)
>   → 디버깅/테스트용
>
> Groth16 verify:
>   검증자는 instance만 알고도 증명을 검증
>   witness는 증명 안에 "암호학적으로 숨겨져" 있음
>   → 이것이 Zero-Knowledge!
>
> 둘의 관계:
>   is_satisfied()가 true인 witness → 유효한 Groth16 proof 생성 가능
>   is_satisfied()가 false인 witness → 유효한 proof 생성 불가능
> ```

### R1CS의 완전성과 건전성

> [!important] R1CS가 올바른 "언어"인 이유
> R1CS 제약 시스템이 유용하려면 두 가지 성질이 필요하다:
> - **완전성(Completeness)**: 올바른 계산은 항상 제약을 만족한다
> - **건전성(Soundness)**: 제약을 만족하면 계산이 실제로 올바르다

```
정리 (완전성 — Completeness):
  올바른 계산의 trace는 항상 R1CS를 만족한다.

  증명:
    함수 f에 대해 y = f(x)가 참이면,
    산술 회로의 각 게이트가 올바르게 계산된 trace를 witness로 사용.

    각 곱셈 게이트 a × b = c에 대해:
      실제 계산에서 c = a × b이므로
      → ⟨Aᵢ, s⟩ × ⟨Bᵢ, s⟩ = a × b = c = ⟨Cᵢ, s⟩ ✓

    모든 제약이 만족됨 → is_satisfied() = true □

정리 (건전성 — Soundness):
  R1CS를 만족하는 witness가 존재하면, 실제로 f(x) = y이다.

  증명:
    witness s = (1, y, x, w₁, ..., wₘ)이 모든 제약을 만족한다고 가정.

    각 제약 ⟨Aᵢ, s⟩ × ⟨Bᵢ, s⟩ = ⟨Cᵢ, s⟩는
    산술 회로의 곱셈 게이트 aᵢ × bᵢ = cᵢ를 강제한다.

    덧셈은 선형결합으로 흡수되어 있으므로 자동으로 보장.

    모든 게이트가 올바르게 계산됨
    → 출력 y = f(x)가 보장됨 □

  주의: 건전성은 "충분히 많은 제약"이 있을 때만 성립.
  제약이 빠지면 잘못된 witness도 통과할 수 있다!

  예시 (불완전한 제약):
    f(x) = x²에서 "y = x · x" 제약을 빠뜨리면
    → s = [1, 99, 3]이 모든 (빈) 제약을 만족
    → y = 99 ≠ 9 = f(3)인데도 통과!

    이것이 "회로를 올바르게 작성하는 것"이 중요한 이유.
    circom 등의 DSL은 컴파일러가 올바른 제약을 자동 생성.
```

> [!note] 건전성 실패의 실제 사례
> ```
> 2022년 Tornado Cash의 circom 회로에서 발견된 버그:
>   제약이 불완전하여 공격자가 유효하지 않은 Merkle proof를
>   생성할 수 있었다.
>
> 교훈:
>   - R1CS 건전성은 "회로 작성자의 책임"
>   - 모든 중간 값에 대한 제약이 빠짐없이 있어야 함
>   - 형식 검증(formal verification)의 중요성
> ```

### R1CS로 검증하는 구체적 수동 계산

> [!abstract] 전체 과정을 숫자로 추적
> R1CS의 동작을 이해하는 가장 좋은 방법은 구체적인 숫자를 넣어
> 처음부터 끝까지 손으로 계산해보는 것이다.

```
예제: x * y = z, where x=3, y=4, z=12

변수 할당:
  Variable::One → 인덱스 0
  z (instance)  → 인덱스 1
  x (witness)   → 인덱스 2
  y (witness)   → 인덱스 3

witness 벡터: s = [1, 12, 3, 4]

제약: x · y = z
  A = [(2, Fr::ONE)]  → "인덱스 2에 계수 1" = x
  B = [(3, Fr::ONE)]  → "인덱스 3에 계수 1" = y
  C = [(1, Fr::ONE)]  → "인덱스 1에 계수 1" = z

선형결합 평가:
  ⟨A, s⟩ = 0·s[0] + 0·s[1] + 1·s[2] + 0·s[3] = 0·1 + 0·12 + 1·3 + 0·4 = 3
  ⟨B, s⟩ = 0·s[0] + 0·s[1] + 0·s[2] + 1·s[3] = 4
  ⟨C, s⟩ = 0·s[0] + 1·s[1] + 0·s[2] + 0·s[3] = 12

검증:
  ⟨A, s⟩ × ⟨B, s⟩ = 3 × 4 = 12 = ⟨C, s⟩  ✓

잘못된 witness: s' = [1, 11, 3, 4]
  ⟨A, s'⟩ × ⟨B, s'⟩ = 3 × 4 = 12 ≠ 11 = ⟨C, s'⟩  ✗
  → is_satisfied() = false
```

> [!tip] 손 계산이 중요한 이유
> ```
> 코드로 is_satisfied()를 호출하면 true/false만 나온다.
> 하지만 숫자를 직접 추적하면:
>
>   1. 어떤 제약이 실패하는지 정확히 알 수 있다
>   2. 선형결합 → 내적 → 곱셈 → 비교의 흐름이 보인다
>   3. 행렬 표현과 LinearCombination 표현의 대응이 명확해진다
>   4. 디버깅 시 "어디서 값이 틀리는지" 추적 가능
>
> 회로를 처음 작성할 때는 반드시 작은 입력으로 손 계산을 해보자!
> ```

to_matrices()는 ConstraintSystem의 제약들을 (A, B, C) 행렬로 변환

출력:

### Witness 생성 알고리즘

> [!important] Witness Generation의 핵심
> "공개 입력 x와 비공개 입력 w로부터 전체 witness 벡터를 계산"하는 과정이다.
> Witness 생성은 곧 **함수를 실제로 실행하는 것**과 같다.

```
Witness Generation:
  "공개 입력 x와 비공개 입력 w로부터 전체 witness 벡터를 계산"

  알고리즘:
    1. s[0] = 1  (상수 One)
    2. s[1..l] = x  (공개 입력)
    3. s[l+1..] = w  (비공개 입력의 원본 값)
    4. 산술 회로를 순서대로 실행하여 보조 변수 채우기

  예 (x³ + x + 5, 입력 x=3):
    s[0] = 1
    s[1] = ?  (y — 아직 모름)
    s[2] = 3  (x)
    s[3] = 3 × 3 = 9  (t₁ = x²)
    s[4] = 9 × 3 = 27 (t₂ = x³)
    s[1] = 27 + 3 + 5 = 35 (y — 이제 계산됨)

    최종: s = [1, 35, 3, 9, 27]

  핵심 인사이트:
    Witness 생성은 "함수를 실제로 실행하는 것"과 같다.
    → 증명자는 f(x)를 실제로 계산할 수 있어야 증명을 만들 수 있다.
    → 검증자는 계산 없이 증명만으로 f(x)=y를 확인!
    → 이것이 "계산의 위임(delegation of computation)"
```

> [!note] 계산의 위임 (Delegation of Computation)
> ```
> 증명자 (Prover):
>   - f(x)를 실제로 계산 → witness 생성 → 증명 생성
>   - 계산 비용: O(|회로|)
>   - 증명 생성 비용: O(|회로| · log|회로|)
>
> 검증자 (Verifier):
>   - f(x)를 계산하지 않음!
>   - 증명만 검증: O(1) 페어링 연산
>   - 공개 입력 x와 결과 y만 알면 됨
>
> 비유:
>   "나는 이 수학 문제를 풀었다" (증명자)
>   "풀이를 보지 않아도, 답이 맞다는 것을 확인할 수 있다" (검증자)
> ```

### R1CS의 한계: 왜 PLONKish가 필요한가

```
R1CS의 제한:
  1. 곱셈 하나 = 제약 하나
     → a + b = c 같은 순수 덧셈도 형식상 (a + b) · 1 = c로 표현
     → 곱셈이 아닌 연산에도 제약을 소모

  2. 커스텀 게이트 없음
     → 모든 제약이 동일한 형태: A · B = C
     → 특수한 연산(range check, lookup)에 비효율적

  3. 와이어 연결이 암묵적
     → "이 변수와 저 변수가 같은 값"을 선형결합으로만 표현
     → 복잡한 라우팅에 보조 변수가 필요

PLONKish (Step 14)는 이 한계를 해결:
  q_L·a + q_R·b + q_O·c + q_M·a·b + q_C = 0

  → 하나의 게이트에 덧셈과 곱셈을 동시에 표현
  → selector (q_L, q_R, ...)로 게이트 종류를 유연하게 선택
  → copy constraint로 와이어 연결을 명시적으로 관리
```

> [!abstract] 세 가지 제약 시스템 비교
>
> | 제약 시스템 | 형태 | 게이트 | Step |
> |-----------|------|--------|------|
> | **R1CS** | A·B = C | 곱셈만 | Step 09 (여기) |
> | **PLONKish** | q_L·a + q_R·b + q_M·ab + ... = 0 | 범용 | Step 14 |
> | **AIR** | P(trace[i], trace[i+1]) = 0 | 행 간 관계 | Step 21 |
>
> R1CS → Groth16 (128B 증명, per-circuit setup)
> PLONKish → PLONK/FFLONK (universal setup)
> AIR → STARK (transparent, post-quantum)

### R1CS에서의 주요 가젯 패턴

> [!abstract] 가젯(Gadget)이란?
> ZK 회로에서 반복적으로 사용되는 R1CS 패턴들을 **가젯**이라 한다.
> 복잡한 회로는 이 기본 가젯들을 조합하여 구성된다.
> Step 10의 Poseidon/Merkle 가젯도 이 패턴들의 조합이다.

```
ZK 회로에서 반복적으로 사용되는 R1CS 패턴들:

1. Boolean 제약 (1개 제약):
   "b는 0 또는 1이다"
   b · (1 - b) = 0

   A = [(b, 1)]
   B = [(One, 1), (b, -1)]
   C = 0

   증명: b=0이면 0·1=0 ✓, b=1이면 1·0=0 ✓
         b=2이면 2·(-1)=-2 ≠ 0 ✗

2. 비트 분해 (254개 제약):
   "x = b₀ + 2b₁ + 4b₂ + ... + 2^253·b₂₅₃"
   각 bᵢ에 대해 boolean 제약: bᵢ·(1-bᵢ) = 0  (254개)
   재조합: x = Σ 2ⁱ·bᵢ (선형결합 — 무료)

   총: 254개 제약
   이것이 비트 연산이 비싼 이유!

3. 범위 검사 (range check):
   "0 ≤ x < 2^n" (n-bit 범위)
   → x를 n비트로 분해 + 각 비트 boolean 제약
   → n개 제약

   예: "잔액이 양수인가?" = range check

4. 등치 제약:
   "x = y"
   → (x - y) · 1 = 0
   → A = [(x, 1), (y, -1)], B = [(One, 1)], C = 0
   → 1개 제약

5. 비영(Non-zero) 제약:
   "x ≠ 0" → "x의 역원 x_inv가 존재"
   → x · x_inv = 1
   → A = [(x, 1)], B = [(x_inv, 1)], C = [(One, 1)]
   → 1개 제약 + 보조 변수 x_inv

   증명자가 x_inv = x⁻¹를 witness로 제공

6. 조건부 선택 (mux):
   result = if b then x else y
   → result = b·x + (1-b)·y = y + b·(x-y)

   보조 변수: t = b·(x-y)
   제약 1: b·(1-b) = 0         (boolean)
   제약 2: b·(x-y) = t         (곱셈)
   최종: result = y + t         (선형결합 — 무료)
   → 2개 제약 (최적화!)

이 패턴들이 Step 10의 Poseidon/Merkle 가젯에서 조합된다.
```

> [!tip] 가젯 조합 예시: Merkle proof 검증
> ```
> Merkle proof 검증은 여러 가젯의 조합:
>   1. Boolean 제약: 경로 방향 (좌/우)
>   2. 조건부 선택: 방향에 따라 해시 입력 순서 결정
>   3. Poseidon 해시: S-box (곱셈 3개) + MDS (무료)
>   4. 등치 제약: 최종 해시 = 공개 root
>
> 각 레벨당: ~250 (Poseidon) + 2 (mux) + 1 (boolean) ≈ 253 제약
> 깊이 20 Merkle tree: ~5,060 제약
> ```

### R1CS → QAP → Groth16 파이프라인

```
전체 흐름:

  Step 09: R1CS (여기)
    함수 f(x) → N개의 제약: (A·s)(B·s) = (C·s)
    "모든 계산을 곱셈으로 분해"
    ↓

  Step 11: QAP (Quadratic Arithmetic Program)
    R1CS 행렬 → 다항식으로 변환
    "N개 제약 검사" → "하나의 다항식 등식" 으로 환원
    핵심: A(x)·B(x) - C(x) = H(x)·Z(x)
    ↓

  Step 12: Groth16
    QAP 다항식을 타원곡선 점으로 인코딩
    페어링(Step 06)으로 검증
    → 증명 크기: 128바이트 (G1 2개 + G2 1개)
    → 검증: 페어링 3~4회

왜 이 변환이 필요한가:
  R1CS: "N개 제약을 하나씩 검사" → O(N) 비용
  QAP: "하나의 다항식 등식" → O(1) 검증 (with 페어링)
  → 이것이 "간결한 증명(succinct proof)"의 핵심!
```

> [!important] Step 09의 위치
> R1CS는 그 자체로 증명 시스템이 아니다.
> R1CS는 **중간 표현(IR)** — 프로그램과 증명 시스템 사이의 다리.
>
> ```
> 프로그램  ←→  R1CS  ←→  QAP  ←→  Groth16
>              [IR]     [수학]    [암호학]
>
> circom, arkworks, bellman 같은 ZK 프레임워크는
> "R1CS를 자동으로 생성하는 DSL"을 제공한다.
> 우리는 수동으로 R1CS를 구성하여 내부 동작을 이해.
> ```

### QAP 변환의 수학적 아이디어

> [!abstract] R1CS → QAP 변환의 핵심 아이디어
> m개의 제약을 각각 검증하면 O(m) 비용이 든다.
> QAP는 이를 **하나의 다항식 등식**으로 환원하여 O(1) 검증을 가능하게 한다.
> 핵심 도구는 **Lagrange 보간**과 **다항식의 나누어떨어짐**이다.

```
R1CS → QAP 변환의 핵심 아이디어:

문제: m개의 제약을 각각 검증 → O(m) 비용
목표: m개의 제약을 하나의 등식으로 환원 → O(1) 검증

핵심 도구: Lagrange 보간 + 다항식의 나누어떨어짐

Step 1: 각 제약에 "평가점" 배정
  제약 1 → ω₁
  제약 2 → ω₂
  ...
  제약 m → ωₘ
  (ω₁, ..., ωₘ은 서로 다른 체 원소)

Step 2: 각 변수 j에 대한 다항식 구성
  Aⱼ(X) = Lagrange 보간으로 구성:
    Aⱼ(ωᵢ) = A[i][j]  (R1CS 행렬 A의 (i,j) 원소)

  마찬가지로 Bⱼ(X), Cⱼ(X) 구성

Step 3: witness 벡터 s로 다항식 결합
  A(X) = Σⱼ sⱼ · Aⱼ(X)
  B(X) = Σⱼ sⱼ · Bⱼ(X)
  C(X) = Σⱼ sⱼ · Cⱼ(X)

핵심 관찰:
  R1CS가 만족됨 ⟺ 모든 i에 대해 A(ωᵢ)·B(ωᵢ) = C(ωᵢ)

  ⟺ 다항식 A(X)·B(X) - C(X)가
     모든 ωᵢ에서 0

  ⟺ A(X)·B(X) - C(X)가
     Z(X) = (X-ω₁)(X-ω₂)···(X-ωₘ)으로 나누어떨어짐

  ⟺ ∃H(X): A(X)·B(X) - C(X) = H(X)·Z(X)

이것이 QAP의 핵심 등식!
  "m개 제약 검증" → "하나의 다항식 나누어떨어짐 검증"

  왜 이것이 효율적인가:
    다항식 등식을 랜덤 점 τ에서 검증 (Schwartz-Zippel):
      A(τ)·B(τ) - C(τ) = H(τ)·Z(τ)

    하나의 점에서의 검증이면 충분! (확률적 보장)
    → Groth16: τ를 "암호학적으로 숨긴 채" 검증 (페어링)

  직관:
    Schwartz-Zippel 보조정리:
      차수 d인 비영 다항식이 랜덤 점에서 0일 확률 ≤ d/|F|
      d ≈ m (제약 수), |F| ≈ 2^254
      → 잘못된 증명이 검증을 통과할 확률 < m/2^254 ≈ 0
```

> [!tip] 왜 다항식인가?
> ```
> 다항식의 핵심 성질:
>   차수 d인 두 다항식이 d+1개 이상의 점에서 같으면 → 동일한 다항식
>
>   이것을 이용:
>     올바른 witness → A(X)·B(X) - C(X)가 Z(X)로 나누어떨어짐
>     잘못된 witness → 나누어떨어지지 않음
>
>   랜덤 점 τ에서 검증:
>     나누어떨어지는데 τ에서 등식이 안 맞을 확률 ≈ 0
>     나누어떨어지지 않는데 τ에서 등식이 맞을 확률 ≈ 0
>
>   → 하나의 점 검증으로 m개 제약을 동시에 검증!
> ```

### Poseidon을 R1CS로: 미리보기 (Step 10)

```
Step 10에서 Poseidon 해시를 R1CS로 변환한다.

S-box: y = x⁵
  t₁ = x · x        ← 제약 1 (x²)
  t₂ = t₁ · t₁      ← 제약 2 (x⁴)
  y  = t₂ · x        ← 제약 3 (x⁵)
  → 3개 제약 / S-box

MDS mix: y = M · state
  y[i] = Σ M[i][j] · state[j]
  → 순수 선형결합 → 제약 0개!

한 라운드:
  AddRoundConstants: 덧셈 → 0개
  S-box (full):     3개 × T = 9개
  S-box (partial):  3개 × 1 = 3개
  MDS mix:          0개

총 Poseidon 제약:
  Full rounds:    8 × 9 = 72
  Partial rounds: 57 × 3 = 171
  합계: 243 ≈ 250 제약

이것이 "ZK-friendly"의 정확한 의미:
  → Poseidon 한 번 = ~250 제약
  → SHA-256 한 번 = ~25,000 제약
  → 100배 차이!
```

### 테스트 전략

```
1. x * y = z (기본 곱셈)
   → 올바른 witness: is_satisfied() = true
   → 잘못된 witness (x*y ≠ z): is_satisfied() = false

2. 피타고라스 x² + y² = z²
   → (3, 4, 5): is_satisfied() = true
   → (3, 4, 6): is_satisfied() = false

3. 잘못된 witness 유형
   → 올바른 구조지만 값이 틀림
   → 제약 시스템이 올바르게 거부하는지 확인

4. to_matrices() 검증
   → 행렬 크기: (제약 수) × (변수 수)
   → 행렬 값이 올바른지 수동 검증

5. Circuit trait
   → synthesize로 제약 생성
   → 다른 witness 값으로 재사용 가능한지 확인
```
