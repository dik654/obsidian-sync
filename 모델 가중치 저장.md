# Step 53: 모델 가중치 저장과 로드

## 개요

Step 51에서 MNIST 학습이 가능해졌지만, 프로그램이 종료되면 학습된 파라미터가 모두 사라진다. Step 53은 학습된 가중치를 **파일로 직렬화(serialize)**하여 저장하고, 다음 실행 시 **역직렬화(deserialize)**하여 복원하는 기능을 구현한다.

이를 통해:
- 학습을 **중단했다 이어서** 할 수 있다
- 학습된 모델을 **배포**할 수 있다 (학습 없이 추론만 수행)
- 다른 환경에서 동일한 모델을 **재현**할 수 있다

---

## 핵심 개념: 직렬화(Serialization)

### 직렬화란?

메모리의 데이터 구조를 바이트 스트림으로 변환하여 파일, 네트워크 등으로 전송 가능하게 만드는 과정:

$$
\mathrm{메모리\ (구조체)} \xrightarrow{\mathrm{serialize}} \mathrm{바이트\ 스트림} \xrightarrow{\mathrm{파일\ 저장}} \mathrm{디스크}
$$

$$
\mathrm{디스크} \xrightarrow{\mathrm{파일\ 읽기}} \mathrm{바이트\ 스트림} \xrightarrow{\mathrm{deserialize}} \mathrm{메모리\ (구조체)}
$$

### 신경망에서 저장해야 할 것

MLP의 학습 가능한 파라미터:

$$
\theta = \{W_1, b_1, W_2, b_2, \ldots, W_L, b_L\}
$$

MLP(784 → 1000 → 10)의 경우:

| 파라미터 | Shape | 원소 수 | 크기 (f64) |
|---|---|---|---|
| $W_1$ | $(784, 1000)$ | $784{,}000$ | 6,272,000 B |
| $b_1$ | $(1000,)$ | $1{,}000$ | 8,000 B |
| $W_2$ | $(1000, 10)$ | $10{,}000$ | 80,000 B |
| $b_2$ | $(10,)$ | $10$ | 80 B |
| **합계** | | **795,010** | **≈ 6.07 MB** |

저장하지 않는 것들:
- **구조 정보** (층 수, 활성화 함수) → 코드에 고정되어 있음
- **기울기** ($\nabla W, \nabla b$) → 매 이터레이션마다 새로 계산
- **옵티마이저 상태** → 단순 SGD는 상태 없음 (Adam 등은 별도 필요)

## 파일 포맷 설계

### Python: NumPy의 .npz 형식

Python DeZero는 `np.savez`를 사용:

```python
model.save_weights('my_mlp.npz')
# → ZIP 아카이브 내부에 각 파라미터가 .npy 파일로 저장
#    my_mlp.npz/
#      ├── 0.npy  (W1: 784×1000 float64)
#      ├── 1.npy  (b1: 1000 float64)
#      ├── 2.npy  (W2: 1000×10 float64)
#      └── 3.npy  (b2: 10 float64)
```

.npy 파일 내부: 매직 넘버 + 헤더(dtype, shape) + raw 데이터.

### Rust: 커스텀 바이너리 형식

추가 의존성 없이 간결한 바이너리 포맷을 설계:

```
┌──────────────────────────────────────┐
│ num_params (u32, little-endian)      │  ← 파라미터 개수 (4바이트)
├──────────────────────────────────────┤
│ [Parameter 0]                        │
│   ndim   (u32)                       │  ← 차원 수
│   shape  (u32 × ndim)               │  ← 각 차원 크기
│   data   (f64 × 원소 수)            │  ← 가중치 값
├──────────────────────────────────────┤
│ [Parameter 1]                        │
│   ndim   (u32)                       │
│   shape  (u32 × ndim)               │
│   data   (f64 × 원소 수)            │
├──────────────────────────────────────┤
│ ...                                  │
└──────────────────────────────────────┘
```

**Little-endian 선택 이유:**
- x86/ARM 모두 little-endian이 기본
- `to_le_bytes()` / `from_le_bytes()`로 변환 비용 없음
- Rust의 관례와 일치

### 구체적 예: $W_1$ (784 × 1000) 저장

```
ndim  = 2                        → 02 00 00 00
shape = [784, 1000]              → 10 03 00 00  E8 03 00 00
data  = [0.0123, -0.0045, ...]   → 각 8바이트 × 784,000개
```

헤더 크기: $4 + 4 \times 2 = 12$ 바이트
데이터 크기: $784{,}000 \times 8 = 6{,}272{,}000$ 바이트

## 파라미터 순서의 결정성

### 순서가 왜 중요한가

저장과 로드가 **같은 순서**로 파라미터를 순회해야 올바른 대응이 보장된다:

```
저장: params()[0] → params()[1] → params()[2] → params()[3]
                                ↓ (파일)
로드: params()[0] ← params()[1] ← params()[2] ← params()[3]
```

### 순서가 결정적인 이유

```rust
// Model::params()의 호출 체인:
fn params(&self) -> Vec<Variable> {
    self.layers()                    // ① Vec<&Linear> — 레이어 순서 고정
        .iter()
        .flat_map(|l| l.params())    // ② 각 레이어: [W, b] 순서 고정
        .collect()
}

// MLP::layers() — Vec의 인덱스 순서대로 반환
fn layers(&self) -> Vec<&Linear> {
    self.layers.iter().collect()      // layers[0], layers[1], ...
}

// Linear::params() — W 먼저, b 다음
pub fn params(&self) -> Vec<Variable> {
    let mut params = Vec::new();
    if let Some(w) = self.w.borrow().as_ref() {
        params.push(w.clone());       // W 먼저
    }
    params.push(self.b.clone());      // b 다음
    return params;
}
```

결과적으로 MLP(784 → 1000 → 10)의 `params()` 순서:

$$
[\underbrace{W_1}_{(784,1000)}, \underbrace{b_1}_{(1000,)}, \underbrace{W_2}_{(1000,10)}, \underbrace{b_2}_{(10,)}]
$$

이 순서는 `layers` Vec의 인덱스에 의해 결정되므로, 같은 MLP 구조라면 항상 동일하다.

## Rc\<RefCell\<\>\>과 set_data의 원리

### 왜 load_weights가 레이어 원본을 갱신하는가

```
Variable은 Rc<RefCell<VarInner>>를 감싸고 있다:

┌─ Linear.w ──────┐      ┌───────────────┐
│ Variable         │─Rc──▶│ VarInner      │
│ (clone of Rc)    │      │   data: [...]  │
└──────────────────┘      │   grad: None   │
                          └───────────────┘
┌─ params()[i] ───┐            ▲
│ Variable         │─Rc────────┘
│ (clone of Rc)    │
└──────────────────┘
```

`params()`가 반환하는 Variable과 레이어 내부의 Variable은 **같은 `Rc`를 공유**한다.
따라서:

```rust
// load_weights 내부에서:
p.set_data(loaded_array);
// → RefCell::borrow_mut() → VarInner.data 교체
// → Linear.w도 같은 VarInner를 가리키므로 자동 갱신
```

이것은 옵티마이저(SGD)가 파라미터를 업데이트하는 것과 동일한 원리다:

```rust
// SGD::update() 내부:
for p in model.params() {
    p.set_data(&p.data() - &grad.mapv(|v| v * self.lr));
    // → 레이어 원본 파라미터도 갱신됨
}
```

## Lazy Initialization과 load_weights의 주의점

### 문제: forward 전에는 W가 없다

Linear의 가중치 $W$는 lazy init으로, 첫 forward 시에 생성된다:

```rust
fn forward(&self, x: &Variable) -> Variable {
    if self.w.borrow().is_none() {
        let in_size = x.shape()[1];   // 입력 차원을 여기서 알게 됨
        *self.w.borrow_mut() = Some(/* Xavier 초기화 */);
    }
    // ...
}
```

따라서 forward를 한 번도 호출하지 않은 상태에서는:

```
params() = [b1, b2]  ← W가 None이라 포함 안 됨 (2개)
파일    = [W1, b1, W2, b2]  ← 저장 시점에는 4개
→ 파라미터 수 불일치 → panic!
```

### 해결: dummy forward로 lazy init 트리거

```rust
// 첫 배치를 꺼내서 forward → W 생성
let (first_x, _) = train_loader.next().unwrap();
let _ = model.forward(&first_x);   // W1(784,1000), W2(1000,10) 생성
train_loader.reset();                // 이터레이터를 처음으로 되돌림

// 이제 params() = [W1, b1, W2, b2] (4개) → 파일과 일치
if Path::new(weight_path).exists() {
    model.load_weights(weight_path);
}
```

## 타이밍과 에폭 측정 (Step 52에서 통합)

Step 52(GPU 모드)는 Rust에서 GPU를 지원하지 않으므로 스킵하고, 에폭 타이밍 개념만 참고로 남긴다:

```rust
// Python: time.time() (wall clock, float 초)
// Rust: std::time::Instant (monotonic clock, 나노초 정밀도)
let start = Instant::now();
// ... 학습 ...
let elapsed = start.elapsed().as_secs_f64();  // Duration → f64 초
```

| | Python `time.time()` | Rust `Instant` |
|---|---|---|
| 시계 종류 | Wall clock | Monotonic clock |
| NTP 영향 | 받음 (시간 역행 가능) | 안 받음 (단조 증가) |
| 정밀도 | 마이크로초 | 나노초 |
| 반환 타입 | `float` | `Duration` |

## Python vs Rust 비교

### save_weights

```python
# Python — np.savez로 한 줄
def save_weights(self, path):
    params_dict = {}
    for i, p in enumerate(self.params()):
        params_dict[str(i)] = p.data
    np.savez(path, **params_dict)
```

```rust
// Rust — 바이너리 직렬화 직접 구현
fn save_weights(&self, path: &str) {
    let params = self.params();
    let mut file = File::create(path).unwrap();
    file.write_all(&(params.len() as u32).to_le_bytes()).unwrap();
    for p in &params {
        let data = p.data();
        // ndim + shape + raw f64 bytes
    }
}
```

### load_weights

```python
# Python
def load_weights(self, path):
    npz = np.load(path)
    for i, p in enumerate(self.params()):
        p.data = npz[str(i)]
```

```rust
// Rust
fn load_weights(&self, path: &str) {
    let bytes = std::fs::read(path).unwrap();
    for p in &self.params() {
        // 바이너리에서 shape + data 역직렬화
        let arr = ArrayD::from_shape_vec(shape, data).unwrap();
        p.set_data(arr);  // Rc 공유로 레이어 원본도 갱신
    }
}
```

| | Python | Rust |
|---|---|---|
| 파일 형식 | .npz (ZIP + .npy) | 커스텀 바이너리 |
| 의존성 | NumPy 내장 | 없음 (std만 사용) |
| 크로스 플랫폼 | .npy가 표준 | little-endian 가정 |
| 파라미터 복원 | `p.data = array` | `p.set_data(array)` (Rc 공유) |
| 구조 검증 | 이름으로 매칭 | 순서로 매칭 |

## 이 Step의 의의

1. **영속성(Persistence)**: 학습 결과를 디스크에 보존 → 프로그램 재시작 후 복원
2. **이어서 학습(Resume Training)**: 중단된 학습을 처음부터 다시 하지 않아도 됨
3. **추론 전용 배포**: 학습 없이 저장된 가중치만 로드하여 예측 수행
4. **직렬화 설계**: 파라미터 순서의 결정성, 바이너리 포맷 설계, `Rc<RefCell<>>` 공유 원리

> 모든 실용적인 딥러닝 프레임워크는 모델 저장/로드를 지원한다:
> PyTorch의 `torch.save`/`torch.load`, TensorFlow의 SavedModel,
> ONNX의 표준 교환 포맷 등. Step 53은 이 기능의 가장 기본적인 구현이다.
